# 单词生成与增量导入 Skill 设计

> 创建日期：2026-07-08
> 状态：待用户审阅

## 1. 背景

当前词库录入流程依赖外部大模型先整理文本，再把结果粘贴到导入页。这个流程存在三个实际问题：

1. 人工整理成本高，录词时需要来回切换工具
2. 模型接入如果直接做进词汇工具，后续更换模型或数据源会把项目代码耦合得越来越重
3. 词库本体当前使用浏览器 IndexedDB 存储，任何“绕过现有导入链路直接写库”的方案都会显著增加维护和排错成本

因此本次选择把“内容生成”能力放到独立 Skill 中，而不是继续把模型能力焊进词汇工具本体。

## 2. 目标

本次设计要实现一条更顺手但边界清晰的录词工作流：

1. 通过固定关键词 `单词：` 触发 Skill
2. Skill 使用“词典 + AI”组合生成标准化单词卡文本
3. 用户先审阅生成结果
4. 用户确认后，Skill 复用现有词汇工具的导入链路完成入库
5. 导入必须是增量导入，不破坏现有词库
6. 模型和词典提供方可替换，尽量不影响词汇工具本体

## 3. 非目标

本次不做以下事情：

1. 不把模型调用直接做进前端项目页面
2. 不让 Skill 直接操作 IndexedDB 底层数据
3. 不实现全量覆盖导入
4. 不做复杂的批量任务编排或后台队列
5. 不在第一版支持多模型 UI 切换
6. 不改变现有学习算法、同步逻辑或备份协议

## 4. 方案选择

本次采用“Skill 负责生成，词汇工具负责导入”的分层方案。

### 4.1 选择该方案的原因

1. 模型可替换性更强：后续更换 Hermes、Trae、火山或其他模型，主要修改 Skill 工作流即可
2. 词汇工具本体保持稳定：继续只负责解析、导入、回滚和存储
3. 出错边界更清晰：生成问题排查 Skill，导入问题排查词汇工具
4. 可以继续复用现有增量导入和回滚能力，而不是另起一套写库逻辑

### 4.2 不采用“把模型直接做进项目”的原因

1. Prompt、错误处理、模型返回结构和兜底策略都会逐渐耦合到项目代码
2. 一旦更换模型或数据源，前端和代理层都可能跟着改动
3. 对个人高频录词场景来说，先用 Skill 验证工作流更轻、更灵活

### 4.3 不采用“Skill 直接写库”的原因

1. 当前词库存在浏览器 IndexedDB 中，脱离浏览器直接写库并不自然
2. 直接写底层存储会绕过现有解析、增量导入和回滚语义
3. 这类方案表面省一步，实际上更脆、更难排错

## 5. 职责边界

### 5.1 Skill 负责

1. 识别 `单词：` 触发输入
2. 解析一个或多个目标单词
3. 查询词典获取基础事实
4. 调用 AI 补齐中文解释层字段
5. 输出严格符合当前导入模板的标准文本
6. 在用户确认后，通过浏览器自动化复用现有导入页完成导入

### 5.2 词汇工具负责

1. 按现有解析规则解析标准文本
2. 按现有增量导入逻辑写入词库
3. 处理重复词覆盖与新增词写入
4. 保留最近一次导入回滚能力

1. 解析规则在 [src/lib/parser.ts](file:///Users/bytedance/Documents/trae_projects/mlbb/vocab-app/src/lib/parser.ts)
2. 增量导入和回滚逻辑在 [src/lib/import.ts](file:///Users/bytedance/Documents/trae_projects/mlbb/vocab-app/src/lib/import.ts)

## 6. 触发与输入设计

### 6.1 触发关键词

Skill 的启动关键词固定为：

```text
单词：
```

示例：

```text
单词：abandon
单词：abandon, ability, absorb
单词：
abandon
ability
absorb
```

### 6.2 批量输入规则

第一版支持单个或少量批量单词：

1. 多个单词可用逗号、空格或换行分隔
2. 第一版单次批量上限为 10 个
3. 超过上限时，Skill 不自动截断，明确提示用户拆分批次

选择 10 个作为上限的原因：

1. 词典查询和 AI 补全时间可控
2. 用户人工确认成本不会过高
3. 失败重试时损失较小

## 7. 数据来源与字段归属

本次明确采用“词典保事实，AI 补中文理解层”的分工。

### 7.1 词典负责字段

1. 音标
2. 词性
3. 中文翻译
4. 英文释义
5. 英文例句

### 7.2 AI 负责字段

1. 释义中文翻译
2. 例句中文翻译
3. 单词记忆技巧

### 7.3 字段优先级规则

1. 词典字段优先视为事实来源，不由 AI 覆盖
2. AI 只补全中文解释层和记忆辅助层字段
3. 若词典缺失英文释义或英文例句，则该单词视为结果不完整，不进入自动导入
4. Skill 必须在确认前显式标出结果不完整的单词

该规则的目标是降低 AI 在英文事实层“看起来对但不够词典级准确”的风险。

## 8. 标准输出格式

Skill 的最终产物是标准文本，而不是结构化 JSON。

原因：

1. 当前项目的文本导入链路已经成熟
2. 标准文本与现有解析器兼容，耦合更低
3. 模型或 Skill 平台变化时，文本协议更稳

输出必须严格匹配当前模板：

```text
【单词】：xxx
【音标（英 / 美）】：/xxx//xxx/
【词性】：xxx
【中文翻译】：xxx
【英文释义】：xxx
【释义中文翻译】：xxx
【英文例句】：xxx
【例句中文翻译】：xxx
【单词记忆技巧】：xxx
```

格式约束：

1. 多个单词之间必须用一个空行分隔
2. 不允许附加解释性前后缀文本
3. 字段名必须与现有解析器识别规则兼容
4. 任一单词缺少必需字段时，Skill 不能把该词混入“可直接导入”结果中

## 9. 用户交互流程

### 9.1 生成阶段

1. 用户输入 `单词：...`
2. Skill 解析单词列表
3. Skill 查询词典并调用 AI 补全字段
4. Skill 输出标准文本预览

### 9.2 确认阶段

1. Skill 输出结果后，不自动导入
2. 用户需要明确发送确认指令后，才进入导入阶段
3. 第一版确认指令固定为：`确认导入`

不采用模糊自然语言确认的原因：

1. “可以”“没问题”“继续”语义不够稳定
2. 导入是写入操作，需要明确边界
3. 固定确认指令更利于 Skill 稳定执行

### 9.3 导入阶段

确认后，Skill 不直接写 IndexedDB，而是：

1. 打开词汇工具的导入页面
2. 将刚刚确认的标准文本填入导入文本框
3. 点击现有“解析并导入”按钮
4. 读取导入结果摘要并反馈给用户

这意味着 Skill 复用的仍是现有文本导入能力，而不是自行实现一套入库逻辑。

## 10. 导入语义

导入必须是增量导入，禁止破坏现有词库。

### 10.1 增量导入定义

采用当前 [importText](file:///Users/bytedance/Documents/trae_projects/mlbb/vocab-app/src/lib/import.ts#L84-L103) 所代表的语义：

1. 新单词：新增到词库
2. 已存在单词：覆盖本次提供的内容字段
3. 其他未涉及单词：保持不变

### 10.2 明确禁止的行为

1. 不允许清空词库后重建
2. 不允许走全量恢复链路
3. 不允许使用会覆盖整个本地库的 [importAll](file:///Users/bytedance/Documents/trae_projects/mlbb/vocab-app/src/lib/backup.ts#L20-L28)

### 10.3 回滚

确认后导入仍然复用现有最近一次回滚机制：

1. 回滚入口和行为不变
2. Skill 不负责实现新的历史版本系统
3. 如果本次导入有误，仍通过现有 [rollbackLastImport](file:///Users/bytedance/Documents/trae_projects/mlbb/vocab-app/src/lib/import.ts#L105-L119) 处理

## 11. 导入桥接设计

由于词库保存在浏览器 IndexedDB 中，Skill 与词库之间的桥接必须经过浏览器。

### 11.1 第一版桥接方案

第一版采用“浏览器自动化桥接”：

1. Skill 在用户确认后导航到目标词汇工具实例的 `/import`
2. 自动填写生成好的标准文本
3. 自动点击导入按钮
4. 读取页面返回结果

### 11.2 目标实例配置

Skill 需要有一个可配置的导入目标 URL：

1. 默认可使用线上部署地址：`https://vocab-app-4zk.pages.dev/import`
2. 若用户平时使用本地开发实例，也可改用本地地址

### 11.3 关键约束

必须明确说明：

1. 不同 origin 的 IndexedDB 彼此隔离
2. 导入到线上 Pages 与导入到本地 `localhost` 是两套独立词库
3. Skill 导入时必须使用用户实际在使用的那一个词库 origin

这是本方案中最容易被误解的点，也是必须写进实现说明的硬约束。

## 12. 错误处理

### 12.1 词典查询失败

1. 某个单词词典查询失败时，Skill 在结果中标记该单词失败
2. 不把失败单词混入可导入结果
3. 用户可以只对成功单词执行确认导入

### 12.2 AI 补全失败

1. 若 AI 失败但词典结果完整，Skill 不自动用空值顶上
2. 缺少中文解释层字段的单词视为未完成
3. 用户可选择重试该词，而不是直接导入不完整条目

### 12.3 导入页自动化失败

1. 若页面无法访问、导入按钮不存在或自动化失败，Skill 需要保留标准文本结果
2. 失败时应提示用户：生成成功，但自动导入失败
3. Skill 需要把本次标准文本再次输出，便于用户手动走导入页

## 13. 第一版范围收敛

第一版只做以下能力：

1. `单词：` 触发
2. 词典 + AI 生成标准文本
3. 明确确认指令 `确认导入`
4. 浏览器自动化复用现有导入页
5. 增量导入，不破坏现有词库

第一版不做以下能力：

1. 多模型选择界面
2. 太大批量的后台任务
3. Skill 直接写底层数据库
4. 复杂冲突处理策略定制
5. 新的历史版本或多级回滚系统

## 14. Skill 结构建议

后续实现该 Skill 时，建议按标准 Skill 结构创建：

1. 目录：`.trae/skills/<skill-name>/`
2. 文件：`.trae/skills/<skill-name>/SKILL.md`

Skill 的描述需要明确两件事：

1. 它做什么：查询词典、调用 AI、输出标准单词文本并支持确认后导入
2. 什么时候触发：当用户输入 `单词：` 形式的单词录入请求时

Skill 名称本身可在实现阶段再最终确认，但设计上应围绕“单词生成与增量导入”这个职责，不要把范围扩散到整个词库管理。
