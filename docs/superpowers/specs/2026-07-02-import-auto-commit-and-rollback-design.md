# 导入自动提交、最近一次回滚与安全区修复设计

> 创建日期：2026-07-02
> 状态：待用户审阅

## 1. 背景

当前项目有两类问题同时存在：

1. 导入页流程偏长：粘贴文本后，需要先“解析”，再逐条处理重复词，最后再“提交”
2. 移动端布局细节不统一：顶部内容容易被 iPhone 刘海区域压住，词库页底部导航视觉位置也和其他页面不一致

这次改动的共同点是都属于“交互壳层”问题：一部分是导入流程壳太重，一部分是页面和导航的安全区壳没统一。

## 2. 目标

本次改动解决五个问题：

1. 导入页点击一次按钮即可完成解析和导入
2. 为最近一次成功导入提供一次性回滚能力
3. 非学习页顶部内容避免被 iPhone 刘海区域遮挡
4. 底部导航在各页面上的视觉位置和可点击区域保持一致
5. 词库列表支持就地删除单词，减少必须进入详情页的操作成本

本次改动不做的事：

1. 不引入多级历史版本
2. 不保留逐条冲突处理 UI
3. 不重做 JSON 备份导入导出协议
4. 不改动学习算法和词库页业务行为
5. 不把全站样式重构成新的设计系统
6. 不新增批量删除或复杂管理模式

## 3. 方案选择

已确认采用“最小 UI 改动”方案：

1. 导入页保留文本框和主按钮
2. 主按钮直接执行“解析并导入”
3. 重复词默认覆盖内容字段，但保留原有学习进度
4. 在导入页新增“回滚最近一次导入”按钮
5. 在页面壳和导航层统一顶部 / 底部安全区处理
6. 在词库列表提供行内删除入口，并保留详情页删除入口

不采用逐条冲突处理的原因：

1. 与“单击完成导入”的目标冲突
2. 现有实现的复杂度主要来自重复词的二次决策
3. 本次已有默认策略：重复词直接覆盖内容字段

不采用“单独微调词库页导航位置”的原因：

1. 问题根源是全局页面壳和导航安全区未统一
2. 单页手调数值会继续制造页面间差异
3. 这类问题适合一次性收敛到共享样式

不采用“管理模式 / 批量删除”的原因：

1. 当前诉求是让词库页可以直接删词，不是做批量整理工作流
2. 项目里已经存在详情页删除能力，本次更适合补齐列表页快捷入口
3. 行内删除 + 确认弹窗已经能覆盖主要使用场景

## 4. 用户可见行为

### 4.1 自动导入

导入页点击主按钮后的流程调整为：

1. 解析文本
2. 查找已存在单词
3. 对新词执行新增
4. 对重复词执行内容覆盖
5. 保存“最近一次导入记录”
6. 清空输入框
7. 显示结果摘要

结果摘要格式保持简单，例如：

`新增 12，覆盖 5`

### 4.2 覆盖规则

重复词导入时：

1. 保留原单词的学习状态字段：`s`、`streak`、`ef`、`interval`、`dueAt`、`reviewedAt`、`createdAt`
2. 更新内容字段：`p`、`pos`、`cn`、`en`、`enCn`、`ex`、`exCn`、`tip`
3. 更新 `updatedAt`

行为与当前手动选择“覆盖”一致，只是默认批量执行。

### 4.3 最近一次回滚

导入页新增“回滚最近一次导入”入口，仅在存在可回滚记录时启用。

点击后执行：

1. 删除这次导入新增的词
2. 恢复这次导入前被覆盖词的旧值
3. 清空最近一次导入记录
4. 显示结果摘要，例如：`已回滚最近一次导入：删除 12，恢复 5`

回滚前需要二次确认，确认文案要明确说明影响范围，例如：

`将删除最近一次导入新增的单词，并恢复被覆盖单词的旧内容。该操作不可再次回滚。`

### 4.4 顶部安全区

非 `Study` 页统一增加顶部安全区留白。

用户可见结果：

1. 首页、词库、导入、统计、设置页标题不再顶到状态栏 / 刘海区域
2. 普通机型和桌面端不会出现夸张的顶部空白

### 4.5 底部导航一致性

底部导航和非学习页页面壳统一收敛。

用户可见结果：

1. 底部导航在 iPhone 上不会贴住屏幕最底部
2. 词库页底部导航的高度和点击体验与首页、导入、统计、设置页一致
3. 非学习页主内容区的底部留白统一，不再出现某一页明显更低的问题

### 4.6 词库行内删除

词库列表每一行新增独立删除按钮，和进入详情的点击区域分离。

点击后流程：

1. 点击行内删除按钮
2. 弹出确认：`确定删除 <word> 吗？`
3. 确认后调用现有删除能力
4. 删除成功后从当前列表状态中移除该词
5. 保持当前 tab 和搜索词不变

保留现有详情页删除入口，形成两种使用场景：

1. 列表页快速整理时，直接行内删除
2. 查看完整信息后，仍可在详情页删除

## 5. 数据设计

### 5.1 IndexedDB 扩展

当前 `meta` store 只有 `settings` 一条记录。本次继续沿用 `meta` store，不新增 object store，只扩展 key 类型。

新增一条 `meta` 记录：

```ts
type MetaKey = 'settings' | 'lastImport'

interface LastImportRecord {
  importedAt: number
  addedWords: string[]
  overwrittenBefore: Word[]
  summary: {
    added: number
    overwritten: number
  }
}
```

设计理由：

1. `addedWords` 只需要保存单词主键，回滚时按主键删除
2. `overwrittenBefore` 直接保存覆盖前完整 `Word` 快照，回滚时可原样恢复
3. 不引入新的 store，迁移成本更低

### 5.2 数据库版本

需要将 IndexedDB 版本从 `1` 升到 `2`。

迁移要求：

1. 已有 `words` 和 `meta` 数据必须保留
2. 旧用户升级后无需手动清库
3. `upgrade` 逻辑要按 `oldVersion` 做兼容判断，避免对已存在 store 重复创建

## 6. 核心实现设计

### 6.1 import 层职责调整

当前 `prepareImport` 和 `commitImport` 是两段式流程。本次保留解析与提交的内部能力，但对页面暴露一个更高层的“自动导入”入口。

建议新增：

```ts
interface ImportSummary {
  added: number
  overwritten: number
}

async function importText(text: string, now?: number): Promise<ImportSummary>
async function rollbackLastImport(): Promise<{ deleted: number; restored: number }>
```

其中：

1. `importText` 负责串联解析、查重、覆盖写入和记录最近一次导入
2. `rollbackLastImport` 负责按最近一次导入记录反向恢复

现有 `prepareImport` / `commitImport` 可根据测试和复用情况保留或收敛，但页面层不再依赖逐条决策状态。

### 6.2 原子性要求

“写入单词”与“保存最近一次导入记录”必须放在同一个 IndexedDB 事务中完成。

原因：

1. 如果单词写入成功但没有写入回滚记录，用户将无法回滚
2. 如果先写入回滚记录但单词未写入完成，会留下错误的可回滚状态

因此数据库层需要提供一个更高层的事务型入口，至少覆盖：

1. 批量写入 `words`
2. 写入或覆盖 `meta.lastImport`
3. 回滚时批量删除 / 批量恢复，并删除 `meta.lastImport`

### 6.3 页面壳与导航统一

当前页面容器存在不一致：有的页使用 `max-w-md`，有的页没有；有的页是 `p-6`，有的页是 `p-4`；导航栏本身也没有消费底部安全区。

本次统一思路：

1. 为非 `Study` 页定义一套共享页面壳样式，统一宽度、水平内边距、顶部留白和底部留白
2. `NavBar` 自身增加底部安全区内边距，并保证点击热区高度不缩小
3. `Study` 页保持独立，不强制复用同一个页面壳

推荐实现位置：

1. 全局样式：放安全区相关 class 或 CSS 变量
2. 页面组件：套统一 class
3. `NavBar.tsx`：只负责自己的固定定位和底部安全区

## 7. 回滚边界

### 7.1 仅支持最近一次成功导入

系统只保留一份 `lastImport` 记录。新的成功导入会覆盖旧记录。

这意味着：

1. 连续导入两次后，只能回滚第二次
2. 回滚一次后，不能再对同一次导入重复回滚

### 7.2 与后续编辑或学习操作的关系

回滚的语义定义为：**把受影响单词恢复到最近一次导入前的状态**。

这意味着如果用户在导入后又对这些词进行了学习或编辑，回滚仍然会恢复到导入前快照，后续改动会被一起覆盖或删除。

这是一个有意接受的边界，原因是：

1. 用户的诉求是修正“最近一次错误导入”
2. 本次只做单级回滚，不扩展为通用历史系统
3. 增加逐词变更冲突检测会显著放大复杂度

为降低误操作风险，回滚按钮旁需要配简短提示，例如：

`建议在发现导入异常后尽快回滚。`

## 8. 错误处理

### 8.1 自动导入

1. 解析失败或未识别出任何条目时，不写库，显示错误信息
2. 某些字段缺失时允许导入，结果摘要仍按成功处理
3. IndexedDB 写入失败时，显示错误信息，不清空输入框

### 8.2 回滚

1. 没有 `lastImport` 记录时，按钮禁用或提示“没有可回滚的最近导入”
2. 回滚事务失败时，保留 `lastImport` 记录，提示失败
3. 成功回滚后清空 `lastImport`，避免重复执行

### 8.3 安全区与页面壳

1. 安全区样式必须对不支持 `env()` 的环境优雅降级
2. 桌面端不因安全区样式导致页面出现多余空白
3. 页面壳统一后，浮动按钮和底栏之间不能互相遮挡

### 8.4 词库删除

1. 删除前必须确认，避免列表页误触
2. 删除失败时保留当前列表状态，并提示失败
3. 删除成功后不重置当前 tab 和搜索条件

## 9. UI 调整范围

本次 UI 调整分为三部分：导入页行为调整、全局安全区 / 页面壳统一，以及词库列表行内删除。不改导航信息架构，也不改 Study 页独立布局。

### 9.1 导入页

导入页调整如下：

1. 移除解析后的逐条列表和单选按钮
2. 主按钮改为执行“解析并导入”
3. 新增“回滚最近一次导入”按钮
4. 保留结果消息区域
5. 导入处理中可临时禁用按钮，避免重复点击

按钮文案有两种可选方式：

1. 保持“解析”，但行为改为直接导入
2. 直接改成“解析并导入”

推荐改成 `解析并导入`，因为行为更准确。

### 9.2 顶部安全区

当前顶部内容在 iPhone 刘海屏上容易过于贴边。本次对非 `Study` 页统一增加顶部安全区留白。

实现要求：

1. 页面顶层容器增加 `env(safe-area-inset-top)` 对应的留白
2. 在安全区之外仍保留一层稳定的基础上边距，避免普通机型顶部太紧
3. `Study` 页保持独立，不强制套用相同页面壳

### 9.3 底部导航与页面壳一致性

当前底部导航本身未处理 iPhone 底部安全区，且各页面使用的容器规格不一致，导致词库页和其他页面相比显得更低、点击不顺手。

本次统一规则：

1. `NavBar` 本身增加底部安全区内边距，点击热区随之抬高
2. 带底栏页面统一使用同一套底部内容留白，避免内容被底栏压住或视觉中心下沉
3. 非 `Study` 页统一页面容器宽度和水平内边距，至少对 `Home`、`Library`、`Import`、`Stats`、`Settings` 保持一致

### 9.4 词库列表行内删除

词库页每行结构调整为：

1. 左侧主体区域继续负责进入详情
2. 右侧增加独立删除按钮
3. 删除按钮使用醒目的危险色，但不喧宾夺主
4. 点击删除按钮时必须阻止触发行点击跳转

推荐交互：

1. 第一层使用浏览器原生确认框，改动最小
2. 删除成功后直接刷新当前列表状态，不额外跳转
3. 详情页原有删除按钮继续保留

## 10. 测试设计

本次按 TDD 补充和调整测试，重点覆盖下面几类行为：

### 10.1 import 单测

1. 自动导入会新增新词并记录 `lastImport`
2. 自动导入会覆盖重复词内容并保留学习进度
3. 新的成功导入会覆盖旧的 `lastImport`
4. 回滚会删除新增词并恢复覆盖前快照
5. 回滚成功后会清空 `lastImport`
6. 无可回滚记录时返回明确结果或抛出受控错误

### 10.2 db 单测

1. `meta.lastImport` 读写正常
2. 数据库升级后原有 `settings` 读取不受影响
3. 清空数据时同时清理回滚记录

### 10.3 页面层测试

如当前测试基线允许，可补导入页行为测试：

1. 点击主按钮后直接调用导入逻辑
2. 有可回滚记录时显示可用按钮
3. 回滚成功后结果消息更新

如当前测试基线允许，也可补页面壳 / 导航相关断言：

1. 非 `Study` 页使用统一页面容器 class
2. `NavBar` 包含底部安全区样式
3. `Study` 页仍然不显示底部导航

如当前测试基线允许，也可补词库页删除行为测试：

1. 点击行内删除按钮不会跳转到详情页
2. 确认删除后会调用删除逻辑并从列表移除
3. 删除后当前 tab 和搜索词保持不变

如果页面层测试成本偏高，本次至少保证 `import`、`db` 层行为单测完整，并对页面结构做人工验证。

## 11. 实施拆分

建议按下面顺序实现：

1. 先为数据层与导入层补失败测试
2. 实现 `lastImport` 存储与回滚事务
3. 实现自动导入高层函数
4. 统一页面壳与导航安全区样式
5. 为词库页补行内删除交互
6. 最后简化导入页 UI

这样可以先把高风险的数据正确性锁住，再处理页面改动。

## 12. 风险与取舍

### 12.1 已接受的取舍

1. 回滚不是历史版本系统，只保留最近一次
2. 回滚会恢复到导入前快照，不保留导入后的后续学习或编辑
3. JSON 备份与 GitHub 同步不携带 `lastImport` 记录，将其视为本地临时状态
4. 安全区和页面壳统一只覆盖非 `Study` 页，不强行统一独立学习页
5. 词库删除先做单条行内删除，不扩展为批量操作

### 12.2 这样取舍的原因

1. 满足当前需求所需改动最小
2. 与现有本地优先架构一致
3. 不会把导入页重新做成复杂工作流
4. 不会把这次任务扩大成全面 UI 重构
5. 能直接复用现有删除能力，风险和改动面都更可控
